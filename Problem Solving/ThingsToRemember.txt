    4. Usage of mutexes - Only 1 reader and 1 writer (std::lock_guard<std::mutex> lock(<mutex_name>))
        a. Coarse-grained locking by putting the lock on the method which locks the entire DS  and no concurrent request on DS.
        b. Fine-grained locking on the object of the DS and no concurrent request to that particular object of DS.
        c. Return the mutex by reference as it is non-copyable and non-moveable
    
    5. Usage of shared mutexes - Multiple readers and only 1 writer 
        a. READER - std::shared_lock<std::shared_mutex> lock(<mutex_name>) and WRITER - std::unique_lock<std::shared_mutex> lock(<mutex_name>);
        b. Very useful when DS internally changes the address of the object during rehashing - Example MAP. To be used when reading and writing to the map.
        c. It can be used along with fine-grained locking where you retrieve the value from the map and release the lock. Enable the fine-grained locking and proceed with modifying the value.
        d. Read PersistentCache.cpp
        e. std::scoped_lock lock(getLock(first), getLock(second)); (std::scoped_lock acquires multiple locks without deadlock )
    
    6. Optimisitic And Pessimistic Atomic Concurrency
        1. Optimisitic Concurrency - 
            a. Add versionControl where the client will provide the lastReadVersion of the object and the system will decide to update the object

            b. Useful when writes are rarely conflicting (giving writes responsibility to User)
            c. Have a look at PersistentCache.cpp
        2. Pessimistic Concurrency -
            a. Fine-grained locking the particular object, read the value and then update the value.
            b. Useful when writes are frequently conflicting (giving writes responsibility to system)

    7. Working with asynchronous thread - Look at Scheduler.cpp/Crawler.cppp/AsyncIterator.cpp/ProducerConsumer.cpp. 
        a. If the worker threads share a state, lock using mutex and add conditions for sleeping.
        b. The conditions of sleeping are  - 
            (1) setup a wakeupTime (relative time) using 'wait_for'
            (2) use predicate conditions in the lambda to wake up the thread - 
                a. There is an extra step that when you change the state of variables, call notify_all to 
                    wake up all threads to re-check the conditons in the lambda. 
                b. If it is true, they will wake up. Else, they will keep on sleeping.
        c. Using wait_for, the thread will wake up after the timeout irrespective of the predicate condition.
        d. So, whenever a thread wakes up, you need to check if they need to abort the loop and proceed to abort.
        e. If they don't need to abort, again check the predicate condition manually. If it fails, ask them to continue the while loop 
           and they will sleep again.
        d. Don't have infinite while loop. Use the variable that lead to abort as a condition to while loop.
        e. Be mindful of using notify_one or notify_all depending upon the requirement. If the requirement is to ask all threads to pause, 
           notify_all seems promising. But if the requirement is to execute an order, notify_one will wake up 1 sleeping thread. If you wake up 
           all threads, they will be competing for the same resource but one will win due to shared mutex.
        e. If you don't have a requirement to pause a worker thread and wants the thread to complete without interuption, you can use promise/future.
           However, once promise/future is set, you can't reset and use it again. If you want the thread to keep on reporting the status, use shared_ptr.
        f. With respect to resources constraint -
            1. Bounded queue (where producers and consumers wait for each other), producer thread pool and consumer thread pool share coordination - 
               consumer requesting producer thread to wake up if the queue is now not full and 
               producer requesting consumer thread to wake up if the queue is now not empty. (ProducerConsumer.cpp)
            2. Database Connections (where consumers are dependent on producers) - If you have 10 connections only, calling acquireConnection()
               act as a consumer which sleeps indefinitely until the queue is not empty. 
               On the other hand, release() act as a producer which pushes into the queue and request consumer to wake up.
            3. [Not Resource Constraint] Job Scheduler (where consumers are dependent on producers) - producers keep on pushing to the 
               shared queue and call notify_all and consumers wake up and consume them.

        a. If delimiter is common, use text.find(delimiter, start)
        b. If delimiter is a set, convert that set to string and use  text.find_first_of(delimiter, start) 
            and that will give you the first occurence of any of the delimiter in the string.
            
        // Step 1: Start at position 0
        size_t start = 0;
        
        // Step 2-5: Loop to find all delimiters
        size_t found = 0;
        while ((found = text.find(delimiter, start)) != std::string::npos) { // Returns the exact position of the delimiter
            // Step 3: Extract text between start and found
            std::string piece = text.substr(start, found - start); // inclusive of start and exclusive of found
            result.push_back(piece);
            
            // Step 4: Move position past delimiter
            start = found + 1;
        }
        
        // Step 6: Don't forget the last piece!
        std::string lastPiece = text.substr(start);
        result.push_back(lastPiece);

GENERAL KNOWLEDGE

1. CPU/RAM CONTEXT SWITCHINHG - 
   a. Timer interrupt fires: Current process (Process A) is interrupted
   b. Save state: OS saves Process A's CPU registers to Process A's PCB in RAM (RAM contains every process's PCB)
   c. Check queue: Scheduler reads the ready queue from RAM to find next process (RAM also contains OS Scheduling Queue)
   d. Select Process B: Scheduler picks Process B from the queue
   e. Load PCB: OS reads Process B's PCB from RAM
   f. Restore registers: CPU registers are loaded with values from Process B's PCB
   g. Set program counter: CPU's program counter is set to the RAM address stored in Process B's PCB (say address 0x00401000)
      (RAM also contains the instruction to execute the task of Process B)
   h. Fetch instruction: CPU fetches the instruction at RAM address 0x00401000
   i. Execute: CPU executes that instruction (which might read/write other data in RAM)
   j. Continue: CPU keeps fetching consecutive instructions from RAM until next interrupt

2. Virtual Machines - 
   1. VMs are cheap as compared to physical servers, provides isolation from malware (leetcode) and rapid provisioning of resources such as CI/CD setup
   2. VM -> Hypervisor —> Host OS —> Hardware (Type2) and VM -> Hypervisor —> Hardware (Type 1 - EC2) --> Execution of a task in VM
   3. Hypervisor creates a file called VM1 or VM2 on physical disk and takes the responsibility of transalting the task being executed in VM 
      to a task to be executed in file VM1 and VM2 by Host OS.
       a. In short, these files VM1 and VM2 are allocated space on disk.
   4. Similar to allocating space on physical disk, Hypervisor also allocates some space in RAM and creates its own PCB.

3. Containers - 
   1. Unlike VM which runs their own OS, container shares host OS and is essentially a process with filesystem space allocated.
   2. For example, its absolute path in container is wrapped by Host OS. So, when we try to open a document in the container,
      the HOST OS just appends as a prefix to the document location to be found in actual disk.
   3. Containers are a process, so, it also have PCB in RAM and get some space allocated.
   4. Unlike VM, where translations done by hypervisor is complex, the translations in containers are not complex in relation.

4. RAW Pointer vs Smart Pointers - 
   a. RAW - explicit deletion by calling 'delete'. Smart - Ref Count increases and once it decreases to zero, it is automatically deleted.
   b. Both of these pointers allocates memory on the heap.

5. Types of Smart Pointers -  
   a. Shared - Multiple owners of the object. You need to decide who can be the owners of the object. When you pass by value, ref count increases
      as owners increases. If the last owner dies, the object is destroyed automatically as ref count is zero.
   b. Unique - Only 1 owner at a time. You need to transfer ownership by pass by value. If the owner dies, object is scrapped.
   c. Weak - Not an owner but an observer to the object. 
      If all owners dies, the object is scrapped so the observer can also see that object is scrapped.
      std::shared_ptr locked = weak_ptr.lock() --> if locked exists, that means a new owner is created from an observer.
